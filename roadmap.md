# TicketGuard 开发路线图

| 战略层级 | 核心任务模块 | 任务优先级 / 核心技术点 | 渐进式开发方案（由易到难） | 团队角色分工 |
|---|---|---|---|---|
| **T0 全员第一步** | 工具链规则确认（比赛机制确认） | 核心开工前提 | 查阅 `skills/` 并让 AI 协助评估：确认比赛对 Google Cloud Agent 的使用是否涵盖 Package，决定是本地进行代码开发（import package）还是在云端平台直接开发。 | **全员第一步**（全员共同确认） |
| **T1 数据与入参** | 获取自己 Feature 的数据（数据源攻关） | 开发生命线 | 每个人根据自己负责的 Feature，摸清并攻关对应的数据和接口，主要盘点三种数据路径：<br>**1.** API 渠道（官方实时数据）<br>**2.** Database 渠道（开源数据库）<br>**3.** Agent 渠道（调用特定 Package / Tools） | **核心数据攻关**（全员参与，各自负责自身特征数据） |
| **T1 数据与入参** | URL 选座 / 价格信息提取 | 核心入参提取 | 分析目标网站 URL 规律，确认是通过参数直接解析（如 Query 字符串）还是通过资源 ID 后台 API 换取，以此直接提取出用户的选座和价格基准。 | **核心数据攻关**（全员参与，各自负责自身入参解析） |
| **T2 核心展示** | 场景一：核心 Demo【安全分析维度】 | Demo 核心演示 | **第 1 层（易）**：既有 / 开源数据库黑名单域名筛选，直接匹配已知钓鱼网站。<br>**第 2 层（难）**：Agent 行为分析。利用规则或安全组件，自动探索并识别该 URL 对应网站的共性风险并上报。 | 安全开发组（分配 2 人） |
| **T2 核心展示** | 场景一：核心 Demo【座位评估维度】 | Demo 核心演示 | **第 1 层（易）**：Prompt 引导评分。将从 URL 提取的数据结合二维场馆图，通过 Prompt 指引 AI 做表面视野判断。<br>**第 2 层（难）**：具体数据建模 / 测算。利用座位数据在后台进行场馆建模，或精准测算实际物理距离与视野角度。<br>*（注：评估必须基于「方位 ➔ 区域 ➔ 具体座位」三层级空间基准）* | 座位视野组（分配 1 人） |
| **T2 核心展示** | 场景一：核心 Demo【价格对比维度】 | Demo 核心演示 | **第 1 层（易）**：直接提取 URL 的当前价格，与标准渠道做基础差价对比。<br>**第 2 层（难）**：（暂未详细讨论，后续将根据数据接口完善渐进式方案） | 价格对比组（分配 1 人） |
| **T3 长远计划** | 场景二：长远附加项【模糊需求到具体推荐】 | 附加项 / 优先级低（Demo 完后考虑） | **第 1 层（易）**：提供官方 List。根据用户模糊需求，调用白名单标准列表直接丢给用户，让用户自行点击跳转。<br>**第 2 层（难）**：Agent 主动寻找。Agent 真正动起来，在全网主动抓取并智能推荐最匹配的门票信息。 | **延后实施**（完成核心 Demo 后全员推进） |
